# Evolution of Todo - Phase II Constitution

## Core Principles

### I. Spec-Driven Development (NON-NEGOTIABLE)
Every feature, component, and API endpoint MUST originate from a written specification in `/specs/`. No implementation begins without an approved spec. Specifications serve as the single source of truth for all development activities.

**Rationale**: Eliminates ambiguity, enables parallel agent work, ensures consistent quality, and provides traceability from requirements to implementation.

### II. Agentic Workflow (7 Specialized Agents)
All implementation work is executed by specialized sub-agents in `.qwen/agents/`. Each agent has明确的 responsibilities:
- **Spec Writer Agent**: Creates and maintains all technical specifications
- **Architecture Planner Agent**: Manages monorepo structure and cross-service integration
- **Database Engineer Agent**: Designs and implements SQLModel schema and Neon DB connection
- **Backend Engineer Agent**: Builds secure FastAPI REST API with JWT middleware
- **Frontend Engineer Agent**: Develops responsive Next.js UI with API client
- **Auth Agent**: Handles secure Better Auth + JWT flow across full stack
- **Integration Tester Agent**: Validates end-to-end flows, user isolation, and API connectivity

**Rationale**: Specialization increases quality, enables parallel development, and ensures deep expertise in each domain.

### III. Test-First Implementation (RED → GREEN → REFACTOR)
Every feature MUST follow TDD cycle:
1. Spec Writer creates specification with acceptance criteria
2. Integration Tester defines test cases
3. Tests are written and MUST fail initially (RED)
4. Implementation is generated by agents
5. Tests MUST pass (GREEN)
6. Code is refactored only after tests pass

**Rationale**: Guarantees correctness, prevents regressions, and ensures every feature is testable by design.

### IV. User Data Isolation (Security First)
Every database query MUST be filtered by authenticated `user_id`. No query bypasses user ownership validation. API endpoints MUST validate JWT tokens before accessing user-specific resources.

**Rationale**: Multi-tenant security requires strict data isolation. Breach prevention is paramount for user trust and hackathon evaluation.

### V. JWT-Based Authentication (Single Source of Truth)
Better Auth with JWT plugin is the exclusive authentication mechanism. JWT tokens are:
- Signed with shared `BETTER_AUTH_SECRET` environment variable
- Included in `Authorization: Bearer <token>` header for all API calls
- Validated by FastAPI middleware before processing requests
- Shared seamlessly between Next.js frontend and FastAPI backend

**Rationale**: Stateless authentication enables scalability, simplifies deployment, and ensures consistent user identity across services.

### VI. Monorepo Separation of Concerns
Clear boundaries between services:
- `frontend/`: Next.js 16+ App Router, TypeScript, Tailwind CSS, shadcn/ui components
- `backend/`: Python FastAPI, SQLModel ORM, REST API endpoints
- `specs/`: All specifications, plans, and tasks
- `.specify/`: Spec-Kit Plus templates and scripts
- `.qwen/`: Agent configurations and workflows

**Rationale**: Separation enables independent scaling, testing, and deployment while maintaining clear ownership.

### VII. Premium UI/UX Standards (Judge-Impressive)
All user-facing interfaces MUST meet professional standards:
- Clean, minimalistic design with premium feel (shadcn/ui, Horizon UI inspired)
- Full light/dark mode support with smooth transitions
- Fully responsive (mobile-first, perfect on all screen sizes)
- Smooth micro-animations (Framer Motion or Tailwind transitions)
- High accessibility (ARIA labels, keyboard navigation, WCAG contrast ratios)
- Professional color palette: Primary teal/indigo accents, neutral grays
- Card-based layouts with subtle shadows, hover effects, rounded corners

**Rationale**: Hackathon evaluation heavily weights UX. Professional polish differentiates winning projects.

## Technology Stack & Architecture

### Approved Technologies
- **Frontend Framework**: Next.js 16+ (App Router) with TypeScript
- **Styling**: Tailwind CSS v4 with shadcn/ui component primitives
- **Backend Framework**: Python FastAPI with SQLModel ORM
- **Database**: Neon Serverless PostgreSQL
- **Authentication**: Better Auth with JWT plugin
- **State Management**: React Server Components (default), Client Components only when necessary
- **Testing**: Integration tests via Integration Tester Agent

### Architecture Pattern
```
User Browser
    ↓
Next.js 16+ Frontend (App Router)
    ↓ (JWT in Authorization header)
FastAPI Backend (JWT Middleware Validation)
    ↓ (user_id filtered queries)
SQLModel ORM
    ↓
Neon Serverless PostgreSQL
```

**Rationale**: Modern, scalable architecture with clear separation, stateless authentication, and serverless database for cost efficiency.

## Security & Data Isolation

### Mandatory Security Controls
1. **JWT Validation**: All `/api/*` endpoints MUST validate JWT tokens via FastAPI dependency injection
2. **User Ownership**: Every SQL query MUST include `WHERE user_id = :current_user_id`
3. **Environment Secrets**: `BETTER_AUTH_SECRET` and `DATABASE_URL` MUST use environment variables, never hardcoded
4. **CORS Configuration**: FastAPI MUST configure CORS to allow only frontend origin
5. **Input Validation**: All API inputs validated via Pydantic models
6. **Error Handling**: Generic error messages to client; detailed logs server-side only

### Data Isolation Enforcement
- Tasks belong exclusively to creating user
- No user can view, modify, or delete another user's tasks
- Database indexes MUST include `user_id` for performance
- API responses MUST NOT leak information about other users' data

**Rationale**: Security breaches are catastrophic. User isolation is the core differentiator from Phase 1 console app.

## Development Workflow & Quality Gates

### Spec-Driven Workflow
1. **Spec Creation**: Spec Writer Agent creates detailed specification in `/specs/<feature>/`
2. **Architecture Review**: Architecture Planner Agent ensures system consistency
3. **Implementation**: Relevant agents (Frontend, Backend, Auth, Database) implement in parallel
4. **Testing**: Integration Tester Agent validates end-to-end functionality
5. **Documentation**: All changes documented in PHR (Prompt History Records)

### Quality Gates (Mandatory)
- ✅ Specification exists before implementation
- ✅ Tests written and failing before code generation
- ✅ All tests passing after implementation
- ✅ PHR created in `history/prompts/` with appropriate routing
- ✅ No TypeScript/Python type errors
- ✅ No console errors in browser
- ✅ Manual testing completed for critical flows

### Prompt History Records (PHR)
Every user prompt and agent response MUST be recorded:
- Constitution changes → `history/prompts/constitution/`
- Feature specs → `history/prompts/<feature-name>/`
- General queries → `history/prompts/general/`

**Rationale**: Traceability enables debugging, onboarding, and hackathon demonstration of agentic workflow.

## Governance & Amendment Process

### Constitution Amendments
This constitution MAY be amended when:
- New principles are discovered through development
- Technology stack requires updates
- Security vulnerabilities necessitate changes
- Hackathon requirements evolve

**Amendment Process**:
1. Propose amendment via user discussion
2. Architecture Planner Agent assesses impact
3. Spec Writer Agent updates constitution
4. Version incremented per semantic versioning
5. PHR created documenting rationale

### Versioning Policy
- **MAJOR** (X.0.0): Backward incompatible changes (principle removal, security model changes)
- **MINOR** (1.X.0): New principles, major feature additions, architecture changes
- **PATCH** (1.0.X): Clarifications, wording improvements, typo fixes

### Compliance Review
All agent outputs MUST comply with this constitution. Non-compliant code MUST be rejected and regenerated.
